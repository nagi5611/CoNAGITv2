## 0. 方針（初回実装で守ること）

1. **常にデプロイ可能な main** … 各フェーズ末で「動作確認できる最小単位」まで仕上げてマージする。
2. **データより先に境界** … S3 の公開経路を塞ぎ、メタデータ API に認可を掛けてからファイル実体を接続する。
3. **フェーズを跨ぐ“先回り”をしない** … 例: サムネイル（キュー・Media Transformations）は、単体 PUT での確実なアップロード完了後に着手する。
4. **本番秘密をリポジトリに置かない** … `.env.example` のキー名のみ。実値はシークレットストア／CI のシークレット。
5. **要件の曖昧な部分は「決定ログ」に残す** … ADR または `docs/decisions/` に 1 ファイル 1 判断で記録する。

---

## 1. フェーズ依存関係（概要）

```text
A 開発基盤 → B AWS 最小構成 → C D1 スキーマ
         ↘（並行可）─────────────────────↗
C → D 認証基盤 → E 管理・所属 → F メタデータ CRUD → G アップロード
G → H フロント統合（MVP 画面）→ I CloudFront 配信
I → J ゴミ箱 → K サムネイル＋キュー → L 編集・監査・プロパティ → M 品質・運用
```

## 2. フェーズ別計画

各フェーズに **目的**・**主要タスク（順序）**・**完了条件（ゲート）**・**依存** を記載する。

---

### フェーズ A — リポジトリと開発基盤

**目的** … 以降の実装を安全に積み上げる土台を固める。

**主要タスク**

1. モノレポまたは `apps/web` + `apps/api`（Workers）などディレクトリ方針を決定し、ルートに README（起動手順の骨子）を置く。
2. `package.json`、TypeScript 厳格化、ESLint、フォーマッタを導入（ルールは既存コミュニティ標準で可）。
3. Wrangler プロジェクト作成（Workers、後から D1 / Queues バインディングを追加）。
4. `npx convex dev` は本プロジェクトでは使用しない（要件は Cloudflare D1）。誤用防止を README に一文。
5. CI（lint / typecheck）を GitHub Actions 等で最小構成。

**完了条件（ゲート）**

- [ ] ローカルで `wrangler dev` が起動し、ヘルス用の `GET /health` が応答する。
- [ ] CI が緑である。

**依存** … なし。

---

### フェーズ B — AWS：S3・IAM・CloudFront の最小構成

**目的** … オブジェクトが誤って公開されない状態で、オリジンからの配信経路を検証する。

**主要タスク**

1. S3 バケット（ブロックパブリックアクセス有効）、バケットポリシーは CloudFront（OAC）からのみ読取可能に限定する設計をドキュメント化し実装。
2. CloudFront ディストリビューション作成、OAC 紐付け、テスト用オブジェクトで `200` を確認。
3. IAM：Workers または中間署名サービスが「プリサインに必要な権限」だけを持つロールを設計（最小権限）。
4. CORS：ブラウザ直 PUT のみ先に狭く許可し、必要ヘッダーを列挙（後で調整可能に）。

**完了条件**

- [ ] パブリック URL 直叩きではオブジェクトを取得できない。
- [ ] CloudFront 経由（署名なしのテスト期間のみ可、本番前に署名必須へ）で取得できる、または署名付き URL で取得できる。

**依存** … フェーズ A。

---

### フェーズ C — D1：スキーマ v1 とマイグレーション運用

**目的** … メタデータの正を D1 に置くためのテーブル群と、変更手順を確立する。

**主要タスク**

1. エンティティ洗い出し：`users`、`sessions`、`groups`、`group_memberships`、`group_leaders`、`projects`、`folders`、`files`、ゴミ箱用（`trash_items` 等）、監査（`audit_events`）を要件に照合して ER 図を 1 枚。
2. インデックス方針（グループ ID、プロジェクト ID、親フォルダ ID、論理削除フラグ等）を先に決める。
3. Wrangler D1 マイグレーションの命名規則とロールバック不能であることの運用注意を README に記載。
4. シード：管理者初期ユーザーの投入手順（本番は手動または一度きりスクリプト）。

**完了条件**

- [ ] `wrangler d1 migrations apply` でローカル／開発用 DB が再現可能。
- [ ] 代表的なクエリ（「ユーザーの所属グループ一覧」等）が説明可能。

**依存** … フェーズ A。

---

### フェーズ D — 認証・セッション・認可ミドルウェア

**目的** … 全 API の共通ゲートを実装する。

**主要タスク**

1. パスワードハッシュ（bcrypt 等）、セッション ID（ランダム）、DB 保存、`HttpOnly` `Secure` `SameSite` 属性の方針決定。
2. `POST /auth/login` / `POST /auth/logout`、セッション検証ミドルウェア。
3. ロール（管理者 / グループリーダー / 一般）を解決するヘルパ（**グループ単位**でリーダーかどうか判定）。
4. 未割り当てユーザーの挙動（一覧空）を API レベルで確認。

**完了条件**

- [ ] セッションなしで保護 API に `401`。
- [ ] 管理者のみが触れるルートが誤って一般に開いていない（ルート単体テスト）。

**依存** … フェーズ C。

---

### フェーズ E — 管理・グループ・メンバー割当

**目的** … 要件の「管理者のみユーザー追加」「リーダーは登録済みから検索付きで割当」を満たす。

**主要タスク**

1. 管理者 API：ユーザー CRUD、グループ CRUD、グループリーダー指名。
2. グループリーダー API：メンバー追加・削除、検索クエリ（D1 `LIKE` または正規化キー列）。
3. 監査の最小：管理操作の append-only ログ（任意だが推奨）。

**完了条件**

- [ ] E2E または手動シナリオで「管理者がユーザーを作る → リーダーが同ユーザーをグループに入れる」が通る。

**依存** … フェーズ D。

---

### フェーズ F — プロジェクト・フォルダ・ファイル（メタデータのみ）

**目的** … S3 を触らずに、名前空間とツリー操作を完成させる。

**主要タスク**

1. プロジェクト作成・一覧・削除（**24 時間ルール**はサーバ側で強制、`Asia/Tokyo` で解釈）。
2. フォルダの作成・移動・削除（他人フォルダ可）。
3. ファイル行の作成（`storage_key` UUID、`display_name`、MIME、サイズ、親フォルダ）※アップロード前の「仮登録」方針か「完了後登録」方針を ADR に記録。

**完了条件**

- [ ] UI なしでも API テストでツリーが組める。
- [ ] 権限外グループのプロジェクト ID を指定した操作が `403`。

**依存** … フェーズ E。

---

### フェーズ G — アップロード（S3 直送）

**目的** … 大容量要件（100MB 境界、マルチパート、並列上限、再試行）を満たす。

**主要タスク**

1. 単一 PUT 用プリサイン URL 発行、`ETag` 返却後に D1 のファイル行を確定。
2. マルチパート：開始・パート URL・完了・中止 API。同一 `uploadId` の再利用方針を実装。
3. クライアント：並列数は環境変数（デフォルト 16）。失敗時 3 回リトライ。
4. アップロード完了後にキューへ「サムネ生成ジョブ」を積む（**フェーズ K で本実装**。フェーズ G では no-op でも可だが、メッセージ形式は固定推奨）。

**完了条件**

- [ ] 100MB 超・未満の両方で成功パスが手元で確認できる。
- [ ] ネットワーク途中切断からの再開が手順書通りに可能。

**依存** … フェーズ B、F。

---

### フェーズ H — フロントエンド（MVP）

**目的** … ブラウザから主要フローが通る。

**主要タスク**

1. 技術選定（例: Vite + React、または Hono JSX + Workers Assets）。**1 案に絞り ADR**。
2. ログイン画面、プロジェクト一覧（複数グループ横断）、プロジェクト内ツリー＋アップロード UI。
3. API ベース URL、Cookie 送信（`credentials: 'include'`）の確認。

**完了条件**

- [ ] 非エンジニアがデモを見て「共有フォルダのイメージ」と言えるレベル。

**依存** … フェーズ D、F、G。

---

### フェーズ I — CloudFront 配信・ダウンロード名・キャッシュ

**目的** … 要件の署名 URL/Cookie、`Content-Disposition`（`filename*`）、キャッシュ整合。

**主要タスク**

1. 署名付き URL または Cookie のどちらを主にするか決定（**両対応でもよいが主経路は 1 つに**）。
2. ダウンロード API が `response-content-disposition` 相当の挙動をクライアントに与える（CloudFront→S3 の経路で可能な方法を選定）。
3. 更新後のキャッシュ対策（イミュータブルキー、短い `max-age`、クエリバージョン等）をファイル種別ごとにルール化。

**完了条件**

- [ ] 日本語ファイル名でブラウザ保存時に意図した名前になる（主要ブラウザで確認）。
- [ ] ファイル更新後、プレビューが古いまま残り続けない。

**依存** … フェーズ B、G。

---

### フェーズ J — ゴミ箱（グループ単位）

**目的** … ソフト削除 1 週間、全員閲覧、管理者のみ空化・完全削除、名前衝突 `(1)` 連番。

**主要タスク**

1. D1 にゴミ箱エントリ（元パス、グループ ID、失効時刻、表示名連番処理）。
2. フォルダ削除の再帰：子を一括でゴミ箱行に移すか、参照で辿るかを実装方針として決定。
3. 期限超過バッチ（Workers Cron またはキュー遅延実行）で物理削除＋S3 削除。
4. 管理者の完全削除・空化 API。

**完了条件**

- [ ] 一般削除後 1 週間以内はリストア可能（要件に「復元」と明記がない場合は「ゴミ箱から戻す」を任意機能として切り分け可。最低限一覧と保持が確認できればよい）。
- [ ] 管理者のみがゴミ箱空化できる。

**依存** … フェーズ F、G。

---

### フェーズ K — サムネイル（Queues + Workers + Transformations）

**目的** … 画像縮小・動画先頭フレーム・失敗時プレースホルダ。

**主要タスク**

1. Queue 定義、プロデューサ（アップロード完了）、コンシューマ（変換→S3 `Put`）。
2. 画像：`cf.image` で取得したレスポンスをサムネキーに保存。
3. 動画：Media Transformations（制約: 公式記載のサイズ・コーデック・Range）。超過時はプレースホルダのみ。
4. デッドレター・最大再試行、冪等性キー（`file_id`）で二重生成を防ぐ。

**完了条件**

- [ ] 代表画像・代表動画でサムネが表示される。失敗時はアイコンにフォールバック。

**依存** … フェーズ G、I（プレビュー URL の決め方による）、Cloudflare 側ゾーン設定（Transformations 有効化）。

---

### フェーズ L — テキスト編集・監査・プロパティ

**目的** … UTF-8 / Shift_JIS、保存時監査、リネーム旧新名、右クリックプロパティ。

**主要タスク**

1. エディタ UI、エンコーディング検出またはユーザー指定、保存 API。
2. `audit_events` への append（保存・リネーム・削除）。
3. プロパティ API（作成者・直近保存者・イベント一覧）。

**完了条件**

- [ ] 要件 9 章・8.5 の最低限を満たす。

**依存** … フェーズ F、H。

---

### フェーズ M — 品質・セキュリティ・運用

**目的** … 初回リリースに耐える状態にする。

**主要タスク**

1. セキュリティレビュー（CSRF、Cookie、CORS、S3 バケットポリシー再確認、依存パッケージ監査）。
2. 観測性：Workers ログ、キュー深度アラート、AWS 請求アラート。
3. バックアップ方針（D1 のエクスポート頻度、S3 バージョニング要否）を文書化。
4. E2E テストのクリティカルパス（ログイン→アップロード→ダウンロード→削除）。

**完了条件**

- [ ] リリースチェックリストにすべてチェックが入る。

**依存** … 全フェーズ。

## 3. 失敗しうる事象と対策一覧

| ID | 分類 | 失敗・リスクの内容 | 想定影響 | 高リスクフェーズ | 予防策 | 発生時の対応・緩和策 | 確認方法 |

| R08 | データ整合 | S3 にのみオブジェクトが残り D1 が失敗（孤児オブジェクト） | コスト・情報残存 | G | マルチパート中止 API、失敗時の Compensating transaction（D1 戻し） | ライフサイクルでプレフィックス掃除、手動削除手順 | S3 インベントリと D1 の差分 |
| R09 | 競合 | 同名ファイル同時作成、移動と削除の競合 | 不整合 | F, J | ユニーク制約（`(project_id, parent_id, display_name)` 等）とエラーハンドリング、楽観ロック | ユーザー向け再試行 UI、衝突時リネーム提案 | 負荷・同時実行テスト |
| R10 | 時間 | プロジェクト 24 時間判定が UTC のみでズレる | 誤削除可否 | F | サーバでは `Asia/Tokyo` で解釈しテストに明示 | データ補正スクリプト、ルール緩和の一時フラグ | 境界日時のユニットテスト |
| R11 | 文字列 | 日本語ファイル名の NFC/NFD 不一致で重複や検索漏れ | 表示・重複 | F, J | 正規化ポリシー（保存時 NFC 等）を決め DB に統一 | 移行スクリプトで正規化 | macOS / Windows 双方でアップロード試験 |
| R12 | 文字列 | Shift_JIS 誤検出で文字化け | データ破壊 | L | ユーザー明示エンコーディング、BOM 扱い、保存前プレビュー | バックアップから復元、監査で巻き戻し | 代表文字コードのゴールデンテスト |
| R13 | アップロード | マルチパート中断でパートが残り続ける | コスト | G | ライフサイクルで未完了アップロードの掃除、クライアントで `AbortMultipartUpload` | AWS コンソール／CLI で一括中止手順 | S3 メトリクス |
| R14 | アップロード | 100MB 境界のオフバイワンで誤った API を呼ぶ | 失敗率上昇 | G | 境界値テスト（99MB, 100MB, 101MB） | 閾値の微調整、UI 表示の明確化 | 自動化テスト |
| R15 | CDN | CloudFront が古いオブジェクトを返し続ける | 誤表示 | I | イミュータブルキー、バージョンクエリ、適切 `Cache-Control` | Invalidation（コストに注意） | 更新直後の ETag 検証 |
| R16 | CDN | 署名付き Cookie とドメイン不一致で配信失敗 | 全滅 | I | ステージング環境でドメイン・パスを先検証 | 設定ロールバック | curl とブラウザ双方 |
| R17 | 外部サービス | Media Transformations が入力要件を満たさず全滅 | サムネなし | K | 事前検証（サイズ・コーデック）、フォールバック必須 | プレースホルダのみ運用に切替 | 代表ファイルのマトリクス試験 |
| R18 | 外部サービス | Transformations ゾーン未設定で Worker が失敗 | キュー滞留 | K | インフラチェックリストに「ゾーン有効化」項目 | 機能フラグでサムネ生成オフ | デプロイ前スクリプト |
| R19 | キュー | メッセージ不正でコンシューマが無限リトライ | コスト・滞留 | K | JSON スキーマ検証、DLQ、最大試行回数 | DLQ から手動リプレイまたは破棄 | キュー深度モニタ |
| R20 | キュー | 冪等性欠如で同一ファイルにサムネ多重生成 | 課金・上書き | K | `file_id` をキーに「既存サムネがあればスキップ」 | 重複オブジェクトの掃除 | S3 リストとログ |
| R21 | DB | D1 マイグレーション失敗で本番と開発が分岐 | デプロイ不能 | C | 本番は必ずバックアップ後に適用、ダウンタイム窓 | `down` 不可なら新マイグレーションで修正 | ステージング先行適用 |
