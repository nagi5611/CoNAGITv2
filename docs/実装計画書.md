## 0. 本書の位置づけと「100点」の定義

### 0.1 位置づけ

本書は **実装だけでなく**、**検証・運用・教育・ドキュメント・リスク管理**まで含めた「初回リリースに至るまでのマスター計画」である。詳細設計（API 一覧・全画面仕様）は別紙とし、本書は **フェーズゲート** と **成果物の抜け漏れ防止** を主目的とする。

### 0.2 「100点」とみなす条件（本プロジェクト内の定義）

次をすべて満たす状態を、本書上の **完成（100点）** と定義する。

| # | 条件 |
|---|------|
| 1 | [要件定義書](./要件定義.md) のスコープ内項目が、**トレーサビリティ表（§3）** のいずれかのフェーズ完了条件またはテストケースに割り当てられている。 |
| 2 | **ステージング**でクリティカルパス E2E が緑、**本番**でスモークが緑。 |
| 3 | **操作マニュアル（一般）**・**管理者マニュアル**・**運用 Runbook**・**障害対応手順**・**リリース／ロールバック手順**がリポジトリまたは社内 Wiki にあり、版付き。 |
| 4 | **リスク登録簿（§7）** がレビュー済みで、ヒューマンエラー・組織プロセスの行が「対策の担当・頻度」まで埋まっている。 |
| 5 | 本番シークレットの**ローテーション手順**と**ブレークグラス**（管理者ロックアウト時）が紙または社内ドキュメントで合意済み。 |
| 6 | 初回リリース後 **30 日間のハイパーケア**（§4 フェーズ P）の当番と連絡経路が決まっている。 |

※「世の中のあらゆるリスクを数学的に完全列挙する」ことは不可能である。§7 は **網羅的に近い登録簿**とし、新規リスクは **R番号を追記**して運用する。

## 2. 体制・意思決定・コミュニケーション

| 役割 | 責務（例） |
|------|------------|
| プロジェクト責任者（1 名） | スコープ変更の承認、本番リリース GO/NO-GO |
| テックリード（1 名） | アーキテクチャ整合、ADR 承認、セキュリティレビュー指揮 |
| 実装担当 | フェーズタスク遂行、自己チェックリスト |
| レビュア（1 名以上） | PR 必須レビュー、リスク登録簿の四半期見直し |
| 運用担当（兼務可） | Runbook 実行、請求モニタ、バックアップ |

**意思決定**: 技術的に不可逆な判断は **`docs/decisions/` に ADR**。口頭のみ禁止。

**定例**（推奨）: 週 1 回 30 分（進捗・リスク・ブロッカー）、フェーズ境界でゲートレビュー 60 分。

## 4. フェーズ計画（A〜P）

依存図（前版から拡張）:

```text
A 開発基盤 → B AWS 最小 → C D1
C → D 認証 → E 管理・所属 → F メタデータ → G アップロード
G → H フロント MVP → I CloudFront → J ゴミ箱 → K サムネ＋キュー → L 編集・監査
L → M 品質・セキュリティ総仕上げ
並行推奨: N（ドキュメント）… A 後半からドラフト着手、M で確定
M 後: O 教育・引き継ぎ → P 本番移行・ハイパーケア
```

以下、各フェーズに **ドキュメント出力（ドラフト/確定）** を明記する。

---

**目的** … 安全に積み上げる土台。

**依存** … なし。

**依存** … 本番 GO。

## 5. 横断要件（全フェーズで意識）

| 項目 | 内容 |
|------|------|
| **フィーチャーフラグ** | サムネ・Media Transformations 等、外部依存はフラグで即オフ可能にする |
| **環境** | `local` / `staging` / `production` の命名・シークレットを分離。**本番操作は二人確認**（手順書に明記） |
| **テスト** | 単体（認可・時刻・文字列）、契約テスト（API）、E2E（クリティカルパス）、**探索的テスト**（各フェーズ末に 1 時間） |
| **ブラウザ** | 要件に無いが、**Chrome / Edge / Safari** の最新版を「動作確認対象」として表にする（Firefox は任意） |
| **アクセシビリティ** | 初回は「キーボードで主要操作可能」を最低ラインとして計画に含める |
| **監視** | Workers エラー率、5xx、キュー深度、D1 エラー、S3/CF 4xx/5xx、請求アラート |

## 6. リリース・ロールバック（抜粋手順）

1. **リリース前**: ステージングで E2E 緑、DB マイグレーションは **先行適用の可否** を判断（ダウンタイム許容なら本番前に適用）。
2. **リリース中**: Workers は Blue-Green に近い **バージョン管理**（Wrangler のバージョン ID）を確認、問題時は前バージョンへ戻す手順を Runbook に記載。
3. **リリース後**: スモーク、30 分間は当番がログ監視。

（詳細は `docs/runbooks/release.md` に委譲—本書では義務化のみ記載）
