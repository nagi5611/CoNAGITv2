# ストレージ共有サービス 要件定義書

**文書バージョン**: 1.76  
**想定読者**: 設計・実装・検証担当者  
**前提技術**: Cloudflare（Workers / D1 / Queues 等）、AWS S3、Amazon CloudFront

---

## 1. 目的・スコープ

### 1.1 目的

会社単位で利用する、限定的なストレージ共有サービスを提供する。ユーザー・グループ・プロジェクト単位でファイルを管理し、Web 上でエクスプローラ風の UI によりフォルダ・ファイルの操作、プレビュー、テキスト編集、監査情報の参照を行う。

### 1.2 スコープ内

- 認証・セッション、ロールに基づく操作権限
- グループ・プロジェクト・フォルダ・ファイルのメタデータ管理（Cloudflare D1）
- オブジェクト実体の保存（単一 AWS S3 バケット、プロジェクト単位プレフィックス）
- 配信・ダウンロード（CloudFront、署名付き URL / 署名付き Cookie）
- アップロード（S3 への直接アップロード、閾値に応じたシングル PUT / マルチパート）
- サムネイル（画像・動画の先頭フレーム。3D はアイコン UI のみ。**非同期キュー**による生成処理）
- ゴミ箱（**グループ単位**。ソフト削除・保持期間・管理者による完全削除・空化）
- 監査ログ（テキスト保存・リネーム。差分は記録しない）

### 1.3 スコープ外（本書時点）

- 実際の Git リポジトリ（clone / push / branch / merge）
- バイナリファイルの内容差分・部分変更位置の記録
- テキストの自動保存

## 2. 用語

| 用語 | 定義 |
|------|------|
| 管理者（アドミニストレーター） | 同一ロール。会社へのユーザー登録・削除、全社的な制約を持つ操作を行う。 |
| グループリーダー（プロジェクトリーダー） | 同一ロール。グループ単位。当該グループの一般ユーザーの割当・解除が可能。 |
| 一般ユーザー | グループに所属し、権限に応じてプロジェクト・ファイル等を操作する。 |
| グループ | 会社に属する作業単位。複数存在する。 |
| プロジェクト | グループ内の作業単位。複数存在する。ストレージの論理境界（S3 プレフィックスの単位）。 |
| 未割り当てユーザー | 会社に登録済みだが、いずれのグループにも所属していないユーザー。 |

## 3. 組織・データモデル

### 3.1 組織

- **テナント**: 会社は 1 つ。会社の下に複数の**グループ**がある。
- **プロジェクト**: 各グループの下に複数のプロジェクトを置く。
- **所属**: 1 ユーザーは複数グループに所属し得る。グループリーダーは当該グループの一般ユーザーの一員でもある前提とする。

### 3.2 データの所有

- **メタデータ（フォルダ階層、表示名、権限、プロジェクト／ゴミ箱状態等）**: Cloudflare D1 を正とする。
- **ファイル実体**: AWS S3。同一バケット内で **プロジェクトごとにプレフィックス**を分け、管理を単純化する。
- **オブジェクトキー**: ストレージ上は **UUID 等の一意キー**とし、ユーザー向け表示名・ダウンロード名は D1 に保持する。

### 3.3 一覧・可視性

- ユーザーは**自分が所属するグループに紐づくプロジェクトのみ**閲覧・操作できる。会社全体の横断一覧は提供しない。
- 複数グループ所属の場合、**所属グループそれぞれのプロジェクトを一つの一覧**で表示できること。
- **未割り当てユーザー**はログイン可能だが、グループ別プロジェクトは表示されない（空に近い状態）。

### 3.4 ゴミ箱のスコープ

- ゴミ箱は**プロジェクト単位ではなく、グループ単位**に配置する。
- **監査用の全社共有ゴミ箱**とはしない（グループごとに論理を分離する）。

## 4. ロールと権限

### 4.1 管理者

- 会社へのユーザーの**追加・削除**（新規ユーザー作成権は管理者のみ）。
- グループリーダーの指名（複数可。詳細は運用で D1 に保持）。
- **ゴミ箱を空にする**、**ゴミ箱内の一部または全部を完全削除**（後述）。**ゴミ箱の空化・ゴミ箱からの削除は管理者のみ**。
- プロジェクト削除: **作成から 24 時間経過後**は、グループリーダーと同様に削除可能（後述ルールに従う完全削除）。
- **通常作業領域（ゴミ箱外）からの削除**について、管理者が実行する削除は**完全削除**（D1 メタデータおよび S3 オブジェクトを遅延なく除去）とする。用語「強制削除」は用いず「**完全削除**」に統一する。
- **ファイルおよびフォルダ**について、**完全削除（即時除去）ができるのは管理者のみ**である（後述のゴミ箱運用と整合）。

### 4.2 グループリーダー

- **管理者が既に会社に登録したユーザー**の一覧から、当該グループへの一般ユーザーを**追加・削除**できる。
- ユーザー選択 UI には**検索窓**を設ける。
- プロジェクト削除: **作成から 24 時間経過後**に削除可能（完全削除）。24 時間以内のルールは「4.3 一般ユーザー」と整合させる。
- **ファイルおよびフォルダ**の削除は、一般ユーザーと同様に**ソフト削除（1 週間のゴミ箱保管）**とする。**完全削除は行わない**（管理者のみが完全削除およびゴミ箱からの除去を行う）。

### 4.3 一般ユーザー（グループ所属）

- 所属グループ内で**プロジェクトの作成**ができる。
- **フォルダの作成・削除・移動**: 全所属ユーザーが可能。**他人が作成したフォルダも**削除・移動の対象になり得る（信頼された利用者前提）。
- **ファイルの削除**: 本人以外がアップロードしたファイルも含め、**一般ユーザーは削除可能**。削除の扱いは **10.1**（ソフト削除）に従う。
- **プロジェクト削除**:
  - **作成から 24 時間以内**: **完全削除**（ゴミ箱を経由しない即時削除でよい旨の要件）。
  - **作成から 24 時間経過後**: 一般ユーザーによるプロジェクト削除は**不可**。削除は**グループリーダーまたは管理者**のみ。
- 基準時刻は D1 上の**プロジェクト作成時刻**。タイムゾーンは**日本（Asia/Tokyo）固定**で解釈・表示する。

### 4.4 未割り当てユーザー

- ログインは可能。プロジェクト等は表示しない（前節）。

## 5. 認証

### 5.1 方式

- **HTTP Basic 認証（ブラウザ組み込み）ではなく**、**フォームで ID・パスワードを受け付け、セッション Cookie** を発行する方式とする。

### 5.2 初期・運用

- 開発・初期運用向けに、管理者用の初期 ID/パスワードは**環境変数**で定義可能とする（例は運用側で設定。コードに固定しない）。
- **初回ログイン時のパスワード強制変更**は本要件では必須としない。初期パスワードは管理者が環境変数等で後から変更可能でよい。

## 6. ストレージ・配信

### 6.1 S3

- **バケットは 1 つ**。キーは **プロジェクト単位のプレフィックス**で論理分離する。
- 実体キーは UUID 等。論理ファイル名（ユーザーが付けた名前、日本語・非 ASCII を含む）は D1 に保持する。

### 6.2 CloudFront

- **オリジンは S3 のみ**とする。
- アクセス制御は **署名付き URL** および **署名付き Cookie** を用いる想定とする。

### 6.3 ダウンロード時のファイル名

- S3 キーが UUID であっても、ダウンロード時は D1 の表示名（例: `image.png`）で保存されること。
- 非 ASCII を含むファイル名には **RFC 5987 の `filename*`** を用いる等、クライアントが元の名前で保存しやすい `Content-Disposition` を付与する。実装詳細は AWS S3 `GetObject` の `response-content-disposition` 等の仕様に従う。

### 6.4 サムネイル

- **画像**: 定解像度のサムネイルを生成し、保存先は**同一 S3 バケット内**（別プレフィックス等）を原則とする（運用単純化）。R2 への配置は必須としない。
- **動画**: **先頭フレーム相当の静止画**をサムネイルとして生成・保存する。
- **3D**: 実ファイルのサムネイル生成は必須としない。**3D らしいアイコン・UI**でよい。
- 生成失敗時: **Place フォルダ**のイメージで、種別に応じたアイコン（画像っぽいアイコン等）を表示する。

#### 6.4.1 非同期キューによる生成

- **画像サムネイル**および**動画サムネイル（先頭フレーム）**の生成は、**Cloudflare Queues（または同等の非同期ジョブ）**を用いて処理する。
- キューを採用する主な理由:
  - **UX**: アップロード完了応答をサムネイル生成の完了まで待たせない。
  - **失敗時**: リトライ・デッドレター等の運用をキュー側で扱いやすい。
  - **大容量・制約外**: 動画が Media Transformations の入力上限を超える等、**フォールバック**（プレースホルダーのみ等）に切り替える時間を確保する。
- ワーカー（コンシューマ）での実処理は **Cloudflare Workers** 上で行い、公式の画像・動画変換能力に依存する（Workers の素の JS だけで ffmpeg 相当を完結させる前提は置かない）。

#### 6.4.2 技術前提（Cloudflare 公式ドキュメントに基づく）

- **画像の縮小・フォーマット変換**: Worker 内の `fetch()` に **`cf.image` オプション**（Image Resizing / Transformations）を指定する方式が利用できる。ゾーンでの機能有効化等の前提がある。
- **動画からの静止画抽出**: **Media Transformations**（`mode=frame` 等）。URL 形式（`/cdn-cgi/media/...`）または **Workers の Media バインディング**（`env.MEDIA.input(stream).output({ mode: "frame", ... })`）が利用できる。
- **制約の例**（実装前に最新の公式ページで再確認すること）: 入力動画サイズ上限（例: 100MB 未満の記載）、長さ上限、推奨コーデック（H.264/AAC 等）、Range 要求など。制約を満たさない場合はサムネイル生成をスキップしプレースホルダーにフォールバックする。
- **課金・製品状態**: Image / Media Transformations の料金・無料枠・GA 状態はリリース時点の **Cloudflare Dashboard および Pricing** を参照する。ドキュメントの更新時差があり得るため、実装マイルストーンごとに再確認する。

出典（参照用 URL）:

- [Transform via Workers（Cloudflare Images）](https://developers.cloudflare.com/images/transform-images/transform-via-workers/)
- [Transform videos（Media Transformations）](https://developers.cloudflare.com/stream/transform-videos/)
- [Bind to Workers API（Media Transformations binding）](https://developers.cloudflare.com/stream/transform-videos/bindings/)

### 6.5 キャッシュ

- CloudFront キャッシュとオブジェクト更新の整合（古いオブジェクトが残る問題）については、**イミュータブルキー**、**Cache-Control**、**無効化**、**キャッシュキーにバージョンを含める**等の一般的手段から実装で選択する。要件として「更新後にユーザーが意図した内容を参照できること」を満たす。

## 7. アップロード

### 7.1 方式

- クライアントから **S3 への直接アップロード**（プリサイン等）を基本とする。

### 7.2 シングル PUT とマルチパート

- **ファイルサイズが 100MB 超**: **マルチパートアップロード**を用いる。
- **100MB 以下**: **シングル PUT** でよい。小さいファイルを無理にマルチパートにしない。

### 7.3 並列数

- ブラウザが並列に進められる**アップロードタスク数の上限**を設ける。
- **デフォルト 16**。値は**環境変数で変更可能**とする。

### 7.4 パートサイズ・制約

- マルチパートのパートサイズ等は **Amazon S3 の制約**（例: パート最小サイズ等）に合わせて実装側で決定する。単一 PUT 可能な最大サイズ（100MB まで）も含め、制約は実装時に公式ドキュメントに基づき確定する。

### 7.5 失敗時の再試行・同一アップロード ID

- 失敗時の再試行は **最大 3 回**まで行う。
- **同一マルチパートアップロード ID** の継続利用を含め、中断・再開が自然に動作するよう実装方針をまとめる（要件として「同一 ID を優先して再利用する」）。

### 7.6 複数ファイル

- 複数ファイルを同時に扱う場合も、上記**並列タスク上限**および各ファイルのサイズに応じたシングル/マルチパート方針に従う。

## 8. UI / 機能

### 8.1 エクスプローラ

- Git や Auto Explorer に近い、**ツリーとコンテンツを融合した**操作感で、フォルダ作成・ファイル作成・アップロード・移動等ができること。

### 8.2 プレビュー

- 画像・動画等のプレビューを提供する（詳細フォーマットは実装フェーズで列挙）。
- 3D はアイコン中心 UI（6.4）。

### 8.3 テキスト編集

- **ブラウザ上のテキストエディタ**で編集する。
- 対象拡張子の例: `.txt`, `.md`, `.csv`, `.py`, `.html` 等、**ソース・マークアップ系も編集対象**に含める。
- **画像・動画・3D 等のバイナリは編集 UI の対象外**とする。

### 8.4 文字コード

- テキスト編集で扱うエンコーディングは **UTF-8** および **Shift_JIS** の**両方**を編集 UI で扱えること（読み込み・保存時の変換方針は実装で定義）。

### 8.5 プロパティ

- ファイルを**右クリック**したときに**プロパティ**を表示できること。
- プロパティでは少なくとも次を参照できること:
  - 関連するユーザー情報（例: 作成者、アップロード者、直近の保存者。会話上の「追加したユーザー」に相当する情報を、保持している範囲で表示する）
  - **変更ログ**（監査ログの該当エントリ）
  - **時刻**
  - **リネームが発生した場合**は、その情報（旧名・新名・ユーザー・時刻）を含めて表示可能であること

## 9. 監査ログ

### 9.1 原則

- **内容差分は記録しない**（テキストの「どの部分が変わったか」も**一切記録しない**）。
- **自動保存は行わない**。テキストの編集監査は**ユーザーが明示的に保存したときのみ**記録する。

### 9.2 テキストファイル

- 「どのユーザーが、いつ、編集（保存）したか」を記録する。
- 対象は**テキスト系**（拡張子ポリシーは 8.3 に準拠）。画像・動画・3D 等のバイナリには編集監査を求めない。

### 9.3 リネーム

- **旧表示名・新表示名の両方**をログに残す。併せて**ユーザー**と**時刻**を記録する。
- ストレージキーは UUID のため、同一論理ファイルの追跡とログの対応が可能であること。

### 9.4 削除

- プロジェクト内で追跡可能な形で、削除に関するイベントも記録方針に含める（詳細は実装設計で定義。差分は不要）。

## 10. ゴミ箱

### 10.0 配置単位

- ゴミ箱は **3.4** に従い、**グループ単位**とする。プロジェクト単位のゴミ箱は設けない。

### 10.1 一般ユーザーおよびグループリーダーによる削除

- **一般ユーザー**および**グループリーダー**がファイル等を削除した場合、**即時完全消去ではなく、1 週間保持**する（ソフト削除）。
- **対象**: **ファイルに限らず、フォルダ削除時は配下の全ファイル・サブフォルダを再帰的に**ゴミ箱扱いとし、**中身ごと**ゴミ箱に移す（後からの管理が容易であることを優先）。

### 10.2 管理者の操作（名称の統一）

- 以前検討していた「強制削除」と「完全削除」は**同一意味**とし、UI・用語は **「完全削除」に統一**する（「強制」という別名は用いない）。

### 10.3 アクセス権

- **ゴミ箱へのアクセスは全ユーザー**が行える。
- **ゴミ箱を空にする**、**ゴミ箱内の一部または全部を削除する**操作は**管理者のみ**。

### 10.4 名前衝突

- ゴミ箱内で表示名が衝突する場合、**`(1)`, `(2)` … の連番サフィックス**で重複を避ける。

### 10.5 表示

- 日本語等 **非 ASCII ファイル名**を正しく表示する（UI・ゴミ箱一覧を含む）。

---

## 11. プロジェクト削除（再掲）

- **作成から 24 時間以内**: **完全削除**（即時でよい）。
- **作成から 24 時間経過後**: **グループリーダーまたは管理者**のみがプロジェクトを削除でき、当該削除も**完全削除**でよい。
- 時刻基準・タイムゾーンは **4.3** に同じ。

---

## 12. 非機能・セキュリティ（高レベル）

- 通信は HTTPS を前提とする。
- セッション Cookie の属性（HttpOnly, Secure, SameSite 等）は実装フェーズで定義する。
- パスワード保存は平文禁止。ハッシュ化・ソルト等は実装標準に従う。
- レート制限・WAF は運用要件に応じて後続で追加可能とする。

---

## 13. コスト・課金の確認観点（抑止のためのチェックリスト）

費用を抑えるため、設計・実装の各マイルストーンで次を**公式の料金ページ・ダッシュボードの使用量**と照合する。

| 確認対象 | チェック内容 |
|----------|----------------|
| **Cloudflare Workers** | リクエスト数、CPU 時間（有料プランの壁）、サブリクエスト数。サムネイル生成をキュー＋コンシューマで回す回数。 |
| **Cloudflare Queues** | メッセージ数、バッチサイズ、コンシューマの実行回数・滞留。再試行ポリシーによる二重計上の有無。 |
| **D1** | 読み書き行数、ストレージ、ワーカーからのクエリ頻度（一覧・監査ログ・プロパティ表示）。 |
| **Image / Media Transformations** | **ユニーク変換回数**や従量課金の単位（公式 Pricing の定義に従う）。サムネイルは「同一ソース＋同一パラメータ」の再計算を避ける（キャッシュ・S3 側に保存済みなら再変換しない等）。 |
| **AWS S3** | ストレージ量、`PUT`/`GET`/`LIST`、マルチパートパート数、ライフサイクル（ゴミ箱 1 週間後の削除）。 |
| **CloudFront** | データ転送アウト、リクエスト数、無効化 API の利用回数（使う場合）。署名付き URL の短命化とキャッシュヒット率のバランス。 |
| **クロスクラウド** | Workers から S3 への `fetch`（エグレス）が課金対象になる場合の有無（Cloudflare のエグレス料金ドキュメントを確認）。 |
| **開発環境** | Wrangler の `remote` 実行や本番に近い API 呼び出しが誤って本番課金に乗らないか。 |

上記の数値目標（月額上限等）はビジネス要件として別紙でもよいが、**どのメトリクスをダッシュボードで週次確認するか**は運用設計で決める。

## 14. 環境変数（要件として列挙）

| 変数名（例） | 用途 |
|--------------|------|
| 管理者初期認証情報 | ID/パスワードの初期値（開発・立ち上げ用。本番は運用で変更） |
| 最大並列アップロード数 | デフォルト 16、変更可能 |

## 16. 参照した対話内容の要約

本書は、以下を反映している。

- Cloudflare D1 と S3 二層、CloudFront 署名 URL/Cookie、単一バケット・プロジェクトプレフィックス、UUID キーと D1 表示名
- ロール統合（管理者＝アドミン、グループリーダー＝プロジェクトリーダー）
- 多グループ所属・一覧、未割り当て時の表示、グループリーダーによる割当（登録済みユーザーのみ＋検索）
- フォーム＋セッション認証
- アップロード・マルチパート・並列数・再試行・同一アップロード ID
- 監査（保存時のみ、差分なし、リネームは旧新名）
- ゴミ箱（**グループ単位**、1 週間、再帰、全員閲覧、管理者のみ空化・削除、完全削除の用語統一、連番衝突解消）
- グループリーダーのファイル／フォルダ削除は**ソフト削除のみ**。**完全削除は管理者のみ**。
- サムネイル（画像・動画）を **Queues 等の非同期キュー**で生成。Cloudflare Images（`cf.image`）および Media Transformations（`mode=frame` 等）を前提とする。
- プロパティ右クリック、UTF-8 と Shift_JIS 編集
- コスト確認は **13** のチェックリストと各サービス公式 Pricing を参照
