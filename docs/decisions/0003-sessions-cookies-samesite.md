# ADR 0003: セッション Cookie・SameSite・CSRF の方針

| 項目 | 内容 |
|------|------|
| ステータス | 提案 |
| 日付 | 2026-05-10 |

## コンテキスト

ブラウザから API を呼ぶ際、セッションは HttpOnly Cookie で運ぶ（`docs/security/sessions.md`）。クロスサイトからのリクエストでセッションが悪用されないよう SameSite 等を検討する必要がある。別案として二重送信 Cookie + CSRF トークン、または SPA でのみ Authorization ヘッダーを使う方式もある。

## 決定

セッション Cookie は **`SameSite=Lax`**・**HttpOnly**・パス `/` とする（実装は `src/auth/cookies.ts`）。API は同一サイトからの利用を主前提とし、**追加の CSRF トークンは初回リリースでは必須としない**。クロスサイトで Cookie を付与したステート変更が必要になった場合は、別 ADR でトークン方式または SameSite=None + Secure の可否を再評価する。

## 結果

### ポジティブ

- 実装・フロントの複雑さを抑え、`Lax` により一般的なクロスサイト POST に Cookie が乗りにくい。

### ネガティブ / リスク

- `Lax` でも全リスクを消せるわけではない。サブドメイン分割や第三者サイトからのフォーム POST など、脅威モデルはリリース前レビューで明示する。
- 同一サイト内 XSS があった場合、Cookie ベースセッションは窃取対象になりうる（CSP・出力エスケープは別途）。

### フォローアップ

- 要件が「別オリジンの埋め込み UI」へ広がったときに本 ADR を見直す。
